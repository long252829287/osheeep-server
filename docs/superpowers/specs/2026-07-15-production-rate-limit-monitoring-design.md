# “小家开饭”生产限流与腾讯云监控设计

更新日期：2026-07-15

## 1. 目标

为 `www.osheeep.com` 的正式 API 增加可预测、可回滚的 Nginx 限流，并通过腾讯云从服务器外部监测后端健康状态，将故障和恢复通知发送到现有隐私联系邮箱。

本次只覆盖入口限流、基础健康探测和邮件告警，不引入应用性能监控、日志上报、短信、电话或新的邮件凭据。

## 2. 已确认决策

- 微信登录 `/api/auth/wechat`：每个来源 IP 每分钟 12 次，允许突发 5 次。
- 其他 `/api/`：每个来源 IP 每分钟 300 次，允许突发 60 次。
- 每个来源 IP 最多 20 个并发连接。
- 超限统一返回 HTTP `429`。
- 告警由腾讯云可观测平台发送邮件，不在服务器保存 163 邮箱授权码。
- 告警接收人和故障联系人复用现有隐私联系邮箱。
- 云拨测每 5 分钟执行一次；连续 2 次失败触发告警，恢复时发送恢复通知。

## 3. Nginx 架构

### 3.1 限流共享区

新增 `/etc/nginx/conf.d/00-osheeep-rate-limit.conf`，在 Nginx `http` 上下文中声明：

- `map $uri $osheeep_auth_limit_key`：只有精确路径 `/api/auth/wechat` 使用来源 IP 作为登录限流键，其他路径的键为空，不参与登录限流。
- `limit_req_zone` 登录区：`12r/m`，共享内存 10 MB。
- `limit_req_zone` 通用 API 区：`300r/m`，共享内存 10 MB。
- `limit_conn_zone` 并发连接区：共享内存 10 MB。

限流键使用 `$binary_remote_addr`，不使用客户端可伪造的 `X-Forwarded-For`。

### 3.2 API 入口

在 `/etc/nginx/conf.d/osheeep.com.conf` 的 `location /api/` 中应用：

- 登录限流区：`burst=5 nodelay`。
- 通用 API 限流区：`burst=60 nodelay`。
- 每 IP 并发连接数：20。
- 请求和连接超限状态码：429。

登录请求同时经过登录限流和通用 API 限流；登录限流更严格。非登录 API 因登录限流键为空，只经过通用 API 和并发连接限制。

保留现有 Spring Boot `127.0.0.1:8080` 反向代理和真实来源 IP 请求头；把配置中已经过时的“Node 3000”注释改为“Spring Boot 8080”，不改变其他站点行为。

## 4. 健康检查入口

在 HTTPS `server` 中新增精确匹配 `location = /healthz`：

- 反向代理到 `http://127.0.0.1:8080/actuator/health`。
- 只公开 Spring Boot 基础健康结果，例如 `{"status":"UP"}`。
- 不公开数据库连接、环境变量、版本、线程或其他 Actuator 端点。
- 关闭该路径的访问日志，保留错误日志。
- 健康入口不依赖小程序登录令牌，腾讯云可直接检查 HTTP 200 和响应中的 `UP`。

## 5. 腾讯云监控

### 5.1 云拨测任务

在腾讯云可观测平台创建“端口性能”HTTPS GET 任务：

- 名称：`小家开饭-生产健康检查`。
- 地址：`https://www.osheeep.com/healthz`。
- 频率：5 分钟。
- 成功条件：HTTP 200，且响应正文包含 `UP`。
- 拨测点：至少选择中国大陆两个不同地域或运营商的可用节点。

### 5.2 告警策略

创建云拨测自定义告警策略：

- 对象限定到上述任务和域名。
- 连续 2 个周期不满足成功条件时触发。
- 恢复后发送恢复通知。
- 通知时间为周一至周日全天。
- 通知渠道仅启用邮件。
- 接收人使用现有隐私联系邮箱；若腾讯云要求验证，管理员必须在邮箱中完成接收确认。

同时为当前 CVM 实例启用基础事件告警，至少覆盖实例 Ping 不可达、磁盘只读和腾讯云底层基础设施故障，并绑定同一通知模板。

## 6. 变更与回滚

实施前分别备份：

- `/etc/nginx/conf.d/osheeep.com.conf`
- 已存在的 `/etc/nginx/conf.d/00-osheeep-rate-limit.conf`（如有）

所有新配置先写入临时文件；只有 `nginx -t` 通过后才替换正式文件并执行 `systemctl reload nginx`。如果平滑重载后健康检查或网站/API 验证失败，立即恢复备份，再次执行 `nginx -t` 后重载。

腾讯云任务和告警策略可在控制台单独停用或删除，不影响服务器运行。不要通过停止正式服务制造告警测试。

## 7. 验证标准

### 7.1 Nginx 与服务

- `nginx -t` 成功。
- Nginx 服务保持 `active`。
- `https://www.osheeep.com/` 返回 200。
- `https://www.osheeep.com/healthz` 返回 200，正文包含 `UP`。
- 未登录访问受保护 API 仍返回预期 401。
- 应用服务保持 `active`，本机 Actuator 返回 `UP`。

### 7.2 限流

- 对 `/api/auth/wechat` 发送不包含微信 code 的空 JSON 请求：限流前返回业务校验 400，超过突发额度后出现 429，不调用微信接口。
- 受控发送最多 65 个未登录通用 API 请求：限流前返回 401，超过通用突发额度后出现 429。
- 验证结束后复扫 Nginx 与应用错误日志，不得出现新的 `ERROR`、异常堆栈或敏感信息。

### 7.3 告警

- 云拨测任务首次执行成功，显示 HTTP 200 且内容校验通过。
- 使用腾讯云通知模板的测试发送功能，不停止生产服务。
- 管理员确认现有联系邮箱收到测试邮件。
- 告警历史或测试记录可在腾讯云控制台查看。

## 8. 完成条件

只有以下条件全部满足，才能在正式提审清单中勾选“限流、告警和故障联系人已确认”：

1. Nginx 限流配置已上线且验证到 429。
2. `/healthz` 从公网返回正确结果。
3. 云拨测任务和两类告警策略已启用。
4. 现有联系邮箱已完成腾讯云验证并收到测试邮件。
5. 故障联系人已明确为该邮箱对应的管理员。
6. 配置、回滚路径和验证结果已写入运维手册及交接记录。
